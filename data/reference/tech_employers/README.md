# Tech Employer Lookup

This directory is the **hand-tagged lookup** that turns free-text donor
employer strings from FEC filings into clean tech-company labels. It is the
single most important cleaning layer in this project, and also the single
biggest source of both false positives and false negatives in the published
numbers.

## What's in here

- **`employer_matches_for_review.csv`**
  A broad candidate list of FEC employer strings that *might* belong to a
  tracked tech company. Generated by running coarse substring matches against
  the itemized-contribution files; most rows need a human decision.
- **`employer_matches_for_review_manual_mar31temp.csv`**
  The current working file that gets consumed by the pipeline. Each row that
  is marked as matched maps one (possibly messy) employer string to one
  canonical company name. Reviewed by hand.

The file names are awkward ("mar31temp") because this started as a one-off
spreadsheet. Will need to update soon.

## How it's used

During `pipeline.load_fec`, every itemized contribution's employer string is
looked up in the manual-review CSV. If there is a match to a tracked company,
the contribution is attributed to that company. If there is no match, the
contribution is simply not counted as tech-linked — it still exists in the raw
FEC data, but this project will not see it.

## Why this is imperfect

The donor employer field on an FEC filing is free text, entered by the
filing committee (usually from what the donor wrote on the contribution form).
Real-world examples from the raw data:

- `GOOGLE`, `GOOGLE LLC`, `GOOGLE INC.`, `Google, Inc.`, `google`, `GOOG`,
  `Alphabet`, `ALPHABET INC`, `ALPHABET/GOOGLE`  — all the same employer
- `META`, `META PLATFORMS`, `META PLATFORMS INC`, `FACEBOOK`, `FACEBOOK INC.`,
  `fb`, `INSTAGRAM` — all the same company, some ambiguous
- `APPLE` — usually Apple Inc., but sometimes refers to unrelated entities
  (Apple Bank, Apple Leisure, a law firm, a personal trainer named Apple)
- `SELF EMPLOYED`, `SELF-EMPLOYED`, `N/A`, blank — unmappable, even if the
  person actually does work in tech

### What this produces

- **False negatives**: a donor who really does work at a tracked tech company
  but wrote their employer in a way the lookup does not yet cover — uncommon
  abbreviations, old names (`FACEBOOK` for someone still at Meta), typos,
  blank fields. Their contribution is missing from the totals.
- **False positives**: a string matched to a tracked company that actually
  refers to something else (a non-tech "Apple", a subsidiary that's been sold
  off). Their contribution is counted toward the wrong company.

There is no way to get perfect accuracy with automation alone. Every new cycle
brings new messy strings.

## What inclusion and exclusion mean

- **Inclusion** in the lookup means "this project currently recognizes this
  employer string as belonging to this tracked tech company." It does not mean
  the company is definitively a tech company in every context — tracked-company
  status is a curatorial choice, not an industry classification.
- **Exclusion** does not mean a company is not tech. It may simply be missing
  from the tracked list, or the employer string variant has not been reviewed
  yet.

Any downstream analysis using these files should describe them as
project-curated and incomplete.

## Why this lives under `data/reference/`

These are small (thousands of rows, not millions) curated files that belong
*with* the code rather than *outside* it, so they live under `data/reference/`
instead of under the large FEC bulk-data tree. They are version-controlled;
the FEC raw files are not.
